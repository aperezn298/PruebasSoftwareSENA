Introducción a las Pruebas de Software

Las pruebas de software son un proceso sistemático de evaluación y verificación de que una aplicación funciona según lo esperado y cumple con los requisitos especificados. Es una actividad crítica en el desarrollo de software que garantiza la calidad y confiabilidad del producto final.

Importante: Las pruebas no pueden probar la ausencia de defectos, solo su presencia. El objetivo es encontrar tantos defectos como sea posible con los recursos disponibles.

Objetivos y Conceptos Fundamentales

  • Encontrar defectos: Identificar errores antes del lanzamiento
  • Verificar requisitos: Confirmar que el software cumple las especificaciones
  • Validar funcionalidad: Asegurar que el software hace lo que el usuario necesita
  • Reducir riesgos: Minimizar fallos en producción y costos asociados
  • Mejorar calidad: Asegurar confiabilidad, usabilidad y rendimiento

Clasificación de Pruebas

Por Ejecución
Pruebas Estáticas

Análisis del código sin ejecutarlo. Incluye revisiones de código, inspecciones y análisis estático.

Pruebas Dinámicas

Ejecución real del código con datos de entrada para observar comportamientos.

Por Enfoque
Pruebas Funcionales

Verifican QUÉ hace el sistema. Se centran en los requisitos funcionales y comportamientos esperados.

Pruebas No Funcionales

Verifican CÓMO funciona el sistema. Incluyen rendimiento, usabilidad, seguridad, etc.

Importancia en el Ciclo de Vida del Software

Análisis y Diseño

Revisión de requisitos, análisis de riesgos, planificación de pruebas.

Desarrollo

Pruebas unitarias, revisiones de código, integración continua.

Integración

Pruebas de integración, sistema, rendimiento y aceptación.

Producción

Monitoreo, pruebas de regresión, mantenimiento.

Errores, Defectos y Fallas

ERROR

Mistake humano durante desarrollo. Causa raíz de problemas.

DEFECTO

Manifestación del error en el código. También llamado "bug".

FALLA

Desviación del comportamiento esperado durante ejecución.

Costos de Detección Tardía: El costo de corregir un defecto aumenta exponencialmente conforme avanza el ciclo de desarrollo. Un defecto que cuesta $1 corregir en desarrollo, puede costar $100 en producción.

Nivel más granular de testing. Verifican el funcionamiento de componentes individuales (funciones, métodos, clases) de forma aislada.
  • Ejecutadas por desarrolladores
  • Rápidas y automatizables

  • Utilizan mocks y stubs

Big Bang

Incremental

  • Pruebas funcionales completas

Estrategias de Prueba

Caja Negra

Sin conocimiento del código interno. Se enfoca en entradas y salidas esperadas.

Técnicas:
  • Partición de equivalencia
  • Análisis de valores límite
  • Tablas de decisión
  • Pruebas de estado
Ventajas:
  • Perspectiva del usuario
  • Independiente de implementación
  • Encuentra funciones faltantes
Caja Blanca

Con conocimiento completo del código. Examina estructura interna y lógica.

Técnicas:
  • Cobertura de sentencias
  • Cobertura de ramas
  • Cobertura de condiciones
  • Cobertura de caminos
Ventajas:
  • Cobertura completa del código
  • Optimización de casos de prueba
  • Identifica código no usado
Caja Gris (Gray Box)

Combinación de ambas estrategias. Conocimiento limitado de la estructura interna combinado con testing funcional. Ideal para pruebas de integración y sistema.

Aplicaciones:
  • Pruebas de regresión
  • Testing de APIs
  • Pruebas de penetración
  • Integración de sistemas

🎯 Principios Fundamentales del Testing

Los 7 Principios ISTQB: Fundamentos universales que guían las mejores prácticas de testing.
Testing muestra la presencia de defectos

Las pruebas pueden mostrar que existen defectos, pero no pueden probar que no los hay.

Testing temprano ahorra tiempo y dinero

Las actividades de testing deben comenzar lo antes posible en el ciclo de vida del software.

Los defectos se agrupan

Un pequeño número de módulos suele contener la mayoría de defectos (Principio de Pareto 80/20).

Testing depende del contexto

Las pruebas se realizan de manera diferente según el contexto (web, móvil, crítico, etc.).

Testing exhaustivo es imposible

Es imposible probar todo. El testing debe enfocarse en riesgos y prioridades.

Cuidado con la paradoja del pesticida

Los mismos casos de prueba ejecutados repetidamente dejan de encontrar nuevos defectos.

Ausencia de errores es una falacia

Encontrar y corregir defectos no ayuda si el sistema es inutilizable o no cumple expectativas.

📊 Técnicas de Diseño de Pruebas

Partición de Equivalencia

Divide los datos de entrada en clases donde todos los valores deberían comportarse de manera similar.

Ejemplo: Campo edad (0-17: menor, 18-65: adulto, 66+: senior)
Análisis de Valores Límite

Se enfoca en probar los valores en los bordes de las clases de equivalencia.

Ejemplo: Probar valores 17, 18, 65, 66 para el ejemplo anterior
Tablas de Decisión

Representan combinaciones de condiciones y sus acciones correspondientes en formato tabular.

Condición R1 R2 R3
Usuario válido ✓ ✓ ✗
Password válido ✓ ✗ -
Acción Login Error Error
Transición de Estados

Modela el comportamiento del sistema como una serie de estados y transiciones.

⚠️ Gestión de Riesgos en Testing

Riesgos del Producto
  • Funcionales: Cálculos incorrectos, flujos de trabajo defectuosos
  • No funcionales: Rendimiento pobre, seguridad vulnerable
  • Operacionales: Instalación compleja, incompatibilidad
Impacto: Directamente afecta a usuarios finales y calidad del software.
Riesgos del Proyecto
  • Organizacionales: Falta de recursos, habilidades insuficientes
  • Técnicos: Herramientas inadecuadas, ambiente inestable
  • Proveedores: Dependencias externas, entregas tardías
Impacto: Afecta la capacidad de ejecutar testing efectivamente.
Testing Basado en Riesgos (RBT)

Enfoque que prioriza los esfuerzos de testing basándose en la evaluación de riesgos. Se concentra más tiempo y recursos en áreas de mayor riesgo.

Pruebas Manuales de Software

Capítulo 2 de 3
Aprende técnicas de testing manual y cómo crear casos de prueba efectivos.

¿Qué son las Pruebas Manuales?

Las pruebas manuales son un proceso donde los testers ejecutan casos de prueba manualmente sin usar herramientas de automatización, simulando el comportamiento del usuario final.

Características Principales

Ventajas
  • Flexibilidad y adaptabilidad
  • Detección de problemas de UX
  • Menores costos iniciales
  • Intuición humana
Desafíos
  • Tiempo y recursos intensivos
  • Propensas a errores humanos
  • Difíciles de repetir exactamente
  • Escalabilidad limitada

Proceso de Pruebas Manuales

1. Análisis de Requisitos

Estudiar la documentación funcional y entender qué se debe probar.

2. Planificación de Pruebas

Definir estrategia, alcance, recursos y cronograma de las pruebas.

3. Diseño de Casos de Prueba

Crear casos de prueba detallados con pasos, datos y resultados esperados.

4. Ejecución de Pruebas

Ejecutar los casos de prueba y documentar los resultados obtenidos.

5. Reporte de Defectos

Documentar y reportar los bugs encontrados con evidencias.

Caso de Prueba: Login de Usuario
ID: TC_LOGIN_001
Título: Verificar login con credenciales válidas

Herramientas Útiles
  • Jira: Gestión de bugs
  • TestRail: Gestión de casos
  • Snagit: Capturas de pantalla
  • BrowserStack: Testing cross-browser

Pruebas Automáticas de Software

¿Qué son las Pruebas Automáticas?

Las pruebas automáticas utilizan herramientas y scripts para ejecutar casos de prueba sin intervención manual, comparando resultados reales con los esperados de forma automática.

Pirámide de Pruebas

E2E / UI Tests Pocas, lentas, costosas
Integration Tests Moderadas, verifican interacciones
Unit Tests Muchas, rápidas, económicas

Frameworks y Herramientas Populares

Para JavaScript
  • Jest: Testing framework completo
  • Cypress: E2E testing moderno
  • Playwright: Cross-browser automation
  • Mocha + Chai: Flexible y expresivo
Para Python
  • pytest: Framework principal
  • Selenium: Automatización web
  • unittest: Biblioteca estándar
  • Robot Framework: Keyword-driven

Ejemplo de Prueba Automatizada

Aquí tienes un ejemplo básico de una prueba unitaria en JavaScript usando Jest:

// calculator.js
function add(a, b) {
    return a + b;
}

function multiply(a, b) {
    return a * b;
}

module.exports = { add, multiply };

// calculator.test.js
const { add, multiply } = require('./calculator');

describe('Calculator Functions', () => {
    test('should add two numbers correctly', () => {
        expect(add(2, 3)).toBe(5);
        expect(add(-1, 1)).toBe(0);
        expect(add(0, 0)).toBe(0);
    });

    test('should multiply two numbers correctly', () => {
        expect(multiply(3, 4)).toBe(12);
        expect(multiply(-2, 5)).toBe(-10);
        expect(multiply(0, 100)).toBe(0);
    });
});

Mejores Prácticas

Arrange: Configura los datos y condiciones iniciales.
Act: Ejecuta la funcionalidad que se está probando.
Assert: Verifica que el resultado sea el esperado.

Cada prueba debe ser independiente y no depender del estado o resultado de otras pruebas. Esto garantiza que las pruebas sean confiables y se puedan ejecutar en cualquier orden.

Los nombres de las pruebas deben ser descriptivos y explicar claramente qué se está probando y cuál es el comportamiento esperado. Esto facilita el mantenimiento y la comprensión.
Métricas Importantes
Code Coverage
Porcentaje de código cubierto por pruebas
Test Execution Time
Tiempo total de ejecución de la suite
Flaky Tests
Pruebas que fallan inconsistentemente
Pass Rate
Porcentaje de pruebas que pasan exitosamente

Las pruebas automáticas son fundamentales en CI/CD. Ejecuta tus pruebas en cada commit para detectar problemas tempranamente.
